# Lô tô: namespace `/loto`, Round model, 2 rooms × 20 seats

Lô tô is a multiplayer Module with **2 fixed rooms**, **max 20 seats**, Socket.IO namespace `/loto` (same JWT middleware pattern as Caro; path still `/socket.io` under `APP_PREFIX`). Lifecycle events use `room:*` + `round:*` inside that namespace (not `game:*` / `match:*` / `hand:*`).

A **Round** is bingo-style: Host prepares a **Ticket pool** of 40 tickets from a static dataset; each Player selects a unique **Ticket**; Host starts only when ≥2 seats are **connected** and all have tickets; server auto-draws numbers 1–90 every `DRAW_MS` (default 8s, env-tunable), pausing while phase is `settling`. **Kinh** is server-validated drawn-only (complete horizontal row of 5 in `drawnNumbers`); first claim only; winner gets `applyPoints` +10; invalid claim → 30s personal cooldown. Marks are client-only. Soft disconnect holds seat `RECONNECT_MS` (60s); expired reconnect returns `SEAT_EXPIRED` (not a silent leave-ok). Explicit leave frees seat; connected seats &lt; 2 mid-Round → Round abort, no points. Full bag with no Kinh → abort draw, no points. After end, phase `waiting` keeps seats and ticket assignments (Host may re-prepare to refresh the pool).

**Wire projections:** `room:list` is lobby-redacted (no ticket matrices / drawn bag). `room:state` is per-socket: full `ticketPool` only in waiting; during playing/settling only that Player’s `myTicket`. Draw updates use `round:number_drawn` deltas without re-shipping the full pool. Portal and game iframe each hold their own Socket.IO client on `/loto`; the iframe must `room:join` after it learns it is seated (via `room:list`) so it enters the Socket.IO room channel and receives `round:number_drawn` (portal join alone does not attach the iframe socket). Number call audio is client-side: each client plays `numbers/{n}.mp3` three times in sequence on `round:number_drawn`. Portal reuses `app` surface; catalog `socketNamespace: '/loto'`.
